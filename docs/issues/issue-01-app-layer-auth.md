# Issue: 应用层认证机制

> 状态：待开发
> 优先级：P2
> 创建日期：2026-03-04

## 问题描述

当前 BLE 连接没有任何安全保护，任何知道 Service UUID 的设备都可以连接并接收数据。需要添加应用层认证机制，确保只有授权设备才能进行数据传输。

## 安全风险

| 风险 | 影响 |
|------|------|
| 未授权设备可连接 | 数据泄露 |
| 未授权设备可接收音频流 | 隐私问题 |
| 无法区分合法/非法客户端 | 安全隐患 |

## 建议方案：应用层认证

### 流程设计

```
iOS App                          ESP32
    │                              │
    │──── BLE 连接成功 ───────────►│
    │                              │
    │──── 写入密钥到 Auth CCC ────►│
    │       "VOICE_MASK_2024"     │
    │                              │ 验证密钥
    │                              │
    │◄─── 验证结果 ────────────────│
    │       0x01 = 成功            │
    │       0x00 = 失败            │
    │                              │
    │    [成功] 启动 hello_timer    │
    │◄─── "hello world" ──────────│
    │                              │
    │    [失败] 拒绝数据传输        │
    │◄─── 断开连接 ────────────────│
```

### GATT 服务扩展

新增 Auth Service：

| 角色 | UUID | 类型 | 说明 |
|------|------|------|------|
| Auth Service | `0000AE10-0000-1000-8000-00805F9B34FB` | Primary Service | 认证服务 |
| Auth Challenge | `0000AE11-0000-1000-8000-00805F9B34FB` | Write | App 写入密钥 |
| Auth Status | `0000AE12-0000-1000-8000-00805F9B34FB` | Read/Notify | 认证状态 |

### 认证密钥方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 硬编码密钥 | 简单 | 需要固件更新才能改密钥 |
| 可配置密钥（NVS存储） | 灵活 | 需要配置接口 |
| 动态挑战码 | 更安全 | 实现复杂 |

**建议采用**：硬编码密钥（MVP），后续扩展为 NVS 存储。

### 错误处理

| 场景 | ESP32 行为 |
|------|-----------|
| 密钥正确 | 返回 0x01，启动 hello_timer |
| 密钥错误 | 返回 0x00，等待重试（最多 3 次） |
| 超过重试次数 | 断开连接 |
| 未认证尝试读取数据 | 返回错误/断开连接 |
