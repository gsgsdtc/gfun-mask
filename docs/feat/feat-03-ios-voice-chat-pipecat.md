# Feat-03: iOS 语音聊天前端 + Pipecat 后端 (STT→LLM→TTS)

> 更新日期：2026-03-10
> 来源：用户需求 — iOS增加一个语音聊天的前端，后端使用pipecat跑通流程。通过esp收音，pipecat需要连接走stt Srt tts
> 优先级：P0（Phase 3 核心功能）

## 1. 功能概述

### 1.1 背景

当前 VoiceMask 系统已完成 Phase 2：ESP32 通过 BLE L2CAP 将麦克风采集的 PCM 音频实时传送至 iOS，iOS 可将音频保存为 WAV 文件并本地播放。

Phase 3 目标是接入云端 AI 语音管道，将设备从"录音播放器"升级为"实时语音聊天助手"：用户对着佩戴设备说话，AI 理解内容后以语音回应，实现真正的对话体验。

### 1.2 目标

1. **iOS 端**：新增语音聊天 UI 页面，支持实时对话状态展示（收音中 / 等待响应 / 播放回复）
2. **后端**：部署 Pipecat 语音管道服务，串联 STT（语音转文字）→ LLM（大语言模型）→ TTS（文字转语音）三个环节
3. **链路打通**：ESP32 收音 → BLE → iOS → Pipecat 后端 → 语音回复 → iOS 播放，全链路跑通

### 1.3 非目标

- 本期不涉及 Opus 编码集成（仍使用 PCM 直传，Phase 4 再切换）
- 本期不涉及唤醒词检测（VAD 打断由用户手动控制）
- 本期不涉及多轮对话历史持久化
- 本期不涉及设备端 TTS 播放（回复音频仅在 iOS 上播放）
- 本期不涉及生产级部署（仅本地/开发环境验证）

## 2. 用户场景

### 场景 1: 发起语音对话

- **角色**：用户（佩戴 VoiceMask 设备）
- **触发**：用户在 iOS App 语音聊天页面点击"开始对话"按钮
- **流程**：
  1. App 向 ESP32 发送录音开始指令
  2. ESP32 开始采集麦克风音频，通过 BLE 实时传送至 iOS
  3. iOS 将音频流转发给 Pipecat 后端
  4. Pipecat 完成 STT 后，将识别文字展示在对话界面
  5. LLM 生成回复文字，展示在对话界面
  6. TTS 合成语音，iOS 播放回复音频
  7. 用户听到 AI 回复
- **结果**：完成一轮语音对话，对话内容以文字形式留存在界面

### 场景 2: 结束当前轮次，发起下一轮

- **角色**：用户
- **触发**：用户点击"停止/发送"按钮（或 ESP32 静音超时）
- **流程**：
  1. iOS 停止向 Pipecat 传送音频，发送结束信号
  2. Pipecat 完成当前轮次处理并返回回复
  3. 界面恢复到"待命"状态，可发起下一轮
- **结果**：当轮对话结束，界面清晰显示等待状态

### 场景 3: 查看对话历史（当前会话内）

- **角色**：用户
- **触发**：查看屏幕上的对话记录
- **流程**：当前会话中所有对话轮次（用户说话内容 + AI 回复文字）按时间顺序展示
- **结果**：用户可回顾本次会话内容

### 场景 4: 连接状态异常

- **角色**：用户
- **触发**：BLE 断连或 Pipecat 服务不可达
- **流程**：界面明确提示当前连接状态（BLE 断连 / 后端不可达），禁用对话功能
- **结果**：用户知晓问题所在，不会误操作

## 3. 验收标准

### 功能验收

- [ ] iOS 新增"语音聊天"标签页（与录音列表并列）
- [ ] 对话界面显示当前状态：待命 / 收音中 / 处理中 / 播放回复
- [ ] STT 识别结果（用户说话内容）以气泡形式显示在对话界面
- [ ] LLM 回复文字以气泡形式显示在对话界面
- [ ] TTS 合成的语音通过 iOS 扬声器播放
- [ ] 对话链路端到端跑通：ESP32 收音 → BLE → iOS → Pipecat → 回复语音播放
- [ ] BLE 未连接时，对话功能不可用并有明确提示
- [ ] Pipecat 服务地址可在 iOS App 内配置（无需重新编译）

### 非功能验收

- [ ] 性能：STT 响应延迟（从停止说话到出现识别文字）< 3 秒
- [ ] 性能：完整一轮对话（STT + LLM + TTS）端到端延迟 < 8 秒
- [ ] 稳定性：连续 5 轮对话不出现崩溃或挂起
- [ ] 兼容：iOS 16+

## 4. 约束条件

- **音频格式约束**：当前 ESP32 输出为 PCM 16kHz 16-bit mono，Pipecat 接入时需兼容此格式（或在 iOS 端转换）
- **传输协议约束**：iOS 与 Pipecat 后端之间的通信协议由 Pipecat 框架决定（通常为 WebSocket/WebRTC），需确认 Pipecat Daily/WebSocket 传输方式
- **BLE 带宽约束**：BLE L2CAP 最大吞吐约 50KB/s（640B × 50fps），音频实时转发不得引入超过 1 帧（20ms）的额外延迟
- **后端依赖**：Pipecat 需部署可访问的服务端（开发阶段允许本地 localhost），STT/LLM/TTS 各自依赖第三方 API（如 Deepgram/OpenAI/ElevenLabs 等）
- **依赖模块**：本 feat 依赖 ble-channel 模块（BLE L2CAP 通道）和 audio-capture 模块（ESP32 PCM 直传）正常运行
