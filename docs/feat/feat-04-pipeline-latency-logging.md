# Feat-04: 语音管道性能监控与对话历史后台

> 类型：Feat
> 日期：2026-03-13
> 状态：待设计
> 来源：内部需求

---

## 1. 背景与目标

### 1.1 背景

当前语音聊天的完整链路（用户说话 → ASR 识别 → LLM 推理 → TTS 合成 → iOS 播放）耗时较长，用户体验有明显延迟感。在着手优化之前，需要先建立可测量的基线数据，并能够回溯历史对话内容与对应的耗时，明确瓶颈所在。

### 1.2 目标

1. 每次语音请求完成后，将对话内容与关键延迟指标持久化到本地数据库
2. 提供一个 Web 后台页面，可以查看历史对话列表及每次对话的详细耗时数据
3. 服务端日志同步输出延迟摘要，便于实时观测

### 1.3 非目标

- 不对任何服务做实际性能优化
- 不需要登录或权限控制
- 不对外暴露 Prometheus / OpenTelemetry 格式的指标接口
- 不在 iOS 客户端做任何延迟统计

---

## 2. 用户与场景

**用户**：开发者

**典型场景**：

1. 开发者进行一次语音交互后，打开 Web 后台（如 `http://localhost:8765/admin`），在列表中看到刚才那条对话记录，包含识别文本、AI 回复和各环节耗时。
2. 开发者调整了 LLM 模型或 TTS 音色后，通过对话历史列表对比前后的延迟数据，判断优化效果。
3. 开发者在终端实时看到每条请求的延迟摘要日志，无需打开浏览器。

## 3. 功能描述

### 3.1 数据存储（SQLite）

每次完整语音请求结束后，将以下数据写入本地 SQLite 数据库：

**对话记录（conversations）**：

| 字段 | 说明 | 备注 |
|------|------|------|
| 唯一标识 | 每次会话的 ID | |
| 创建时间 | 请求完成时间 | |
| 用户语音文本 | ASR 识别结果 | |
| AI 回复文本 | LLM 生成的完整回复 | |
| ASR 首包时间（ms） | 从 `stop` 到收到第一个识别片段 | 当前批量模式下与 ASR 耗时相同，流式 ASR 后可独立测量 |
| ASR 耗时（ms） | 从 `stop` 到 STT 返回完整识别结果 | |
| LLM TTFT（ms） | 从 STT 结果送入 LLM 到收到第一个 Token | 流式输出，可测首包 |
| LLM 完整耗时（ms） | 从 STT 结果送入 LLM 到生成完毕 | |
| TTS 首包时间（ms） | 从文本送入 TTS 到收到第一帧音频 | |
| TTS 完整合成时间（ms） | 从文本送入 TTS 到所有音频帧生成完毕 | |
| 整体首包时间（ms） | 从用户发出 `stop` 到服务端发出第一帧 TTS 音频 | = ASR + LLM TTFT + TTS TTFA |

> **说明**：ASR 当前使用 DashScope Paraformer 批量识别接口（非流式），首包时间与总耗时相同，字段保留以便后续切换流式 ASR 后直接生效。LLM 使用 OpenAI 兼容流式接口，可精确测量首 Token 到达时间（TTFT）。

数据库文件存放在 `pipecat-server/` 目录下，随服务启动自动创建。

### 3.2 服务端日志

每次请求结束后，同步输出一行 INFO 级别的延迟摘要到终端和日志文件：

```
[Latency] ASR=320ms | LLM_TTFT=480ms | LLM_total=1200ms | TTS_TTFA=210ms | TTS_total=890ms | E2E_first_audio=1010ms
```

若任一环节耗时超过 1000ms，以 `WARNING` 级别额外输出提示。

### 3.3 Web 后台（集成到 pipecat-server）

在现有 FastAPI 服务中新增 `/admin` 路由，无需独立启动，无需登录。

采用经典管理后台布局，为后续新增管理功能（如配置管理、设备管理等）预留扩展位。

**整体布局**：

```
┌─────────────────────────────────────────────────────┐
│  VoiceMask Admin          [顶部导航栏]               │
├──────────┬──────────────────────────────────────────┤
│          │                                          │
│  [侧边栏] │  [主内容区]                              │
│          │                                          │
│  📊 概览  │                                          │
│  💬 对话  │                                          │
│  ──────  │                                          │
│  （预留） │                                          │
│          │                                          │
└──────────┴──────────────────────────────────────────┘
```

**概览页（`/admin`）**：

作为后台首页，展示核心汇总数据：

| 展示内容 | 说明 |
|----------|------|
| 今日对话次数 | 当天请求总数 |
| 平均整体首包时间 | 近 N 条的平均值 |
| ASR / LLM / TTS 各环节平均耗时 | 性能趋势一览 |
| 最近对话列表（简表） | 最近 5 条，点击跳转详情 |

**对话列表页（`/admin/conversations`）**：

| 展示内容 | 说明 |
|----------|------|
| 时间 | 对话发生时间 |
| 用户说了什么 | ASR 识别文本（截断显示） |
| AI 回复了什么 | LLM 回复文本（截断显示） |
| 整体首包时间 | 用户可感知的核心延迟指标 |
| 操作 | 点击查看详情 |

按时间倒序排列，支持分页（每页 20 条）。

**对话详情页（`/admin/conversations/{id}`）**：

| 展示内容 | 说明 |
|----------|------|
| 完整对话内容 | 用户文本 + AI 回复全文 |
| 耗时分解面板 | ASR 首包 / ASR 总计 / LLM TTFT / LLM 总计 / TTS 首包 / TTS 总计 / 整体首包，每项单独展示（ms） |
| 时间轴条形图 | 各环节耗时横向堆叠，直观显示各阶段占比 |

**侧边栏导航**（预留扩展位）：

| 菜单项 | 路由 | 说明 |
|--------|------|------|
| 概览 | `/admin` | 当前 feat 实现 |
| 对话记录 | `/admin/conversations` | 当前 feat 实现 |
| （预留） | — | 后续功能模块挂载点 |

## 4. 约束与依赖

| 约束 | 说明 |
|------|------|
| 不影响现有功能 | 计时与存储逻辑只做旁路观测，不改变数据流和处理逻辑 |
| 最小侵入 | 计时代码集中管理，避免散落在各处 |
| SQLite | 本地文件数据库，不依赖外部服务 |
| 集成到 pipecat-server | Web 后台复用现有 FastAPI 服务，不新增独立进程 |
| 无鉴权 | 仅限本地开发使用，不暴露到公网 |

**依赖模块**：
- `pipecat-pipeline`（`pipecat-server/`）：添加计时埋点和数据库写入的目标模块

---

## 5. 验收标准

1. 完成一次语音交互后，SQLite 数据库中存在对应记录，包含对话内容和全部延迟字段
2. 访问 `http://localhost:8765/admin` 可看到概览页（汇总数据 + 最近对话）；访问 `/admin/conversations` 可看到分页对话列表；点击任一条目可查看详情和耗时时间轴
3. 日志中每次请求后输出延迟摘要行
4. 计时与存储逻辑不引入任何功能异常（音频播放正常，指令不丢失）
5. 现有测试套件全部通过
