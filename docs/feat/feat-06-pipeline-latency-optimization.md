# Feat-06: 语音管道首包时间优化

> 类型：Feat（性能优化）
> 日期：2026-03-15
> 状态：待设计
> 来源：内部性能观测（feat-04 监控数据）

---

## 1. 背景与目标

### 1.1 背景

Feat-04 建立了端到端的延迟监控体系，已可从日志和 Admin 后台获得完整的耗时数据。首次实测结果如下：

```
ASR_ttfa=951ms | LLM_TTFT=5475ms | TTS_ttfa=830ms | E2E_ttfa=7257ms
```

端到端首包时间（E2E_ttfa）达到 **7257ms**，远超人机对话可接受的流畅阈值（约 1500ms 以内），用户体验有明显的"卡顿感"。

**瓶颈定位**：

| 环节 | 耗时 | 占比 | 说明 |
|------|------|------|------|
| ASR（Paraformer） | 951ms | 13% | 批量识别，无流式优化空间 |
| LLM TTFT（Qwen） | 5475ms | **75%** | **主瓶颈** |
| TTS 首包（CosyVoice） | 830ms | 11% | 可与 LLM 流水线衔接 |

LLM 首 Token 时间（TTFT）占总延迟的 75%，是当前最主要的优化目标。

### 1.2 目标

1. 将端到端首包时间（E2E_ttfa）从 ~7257ms 降低至 **2000ms 以内**
2. 重点降低 LLM TTFT，目标 **≤ 1000ms**
3. 优化过程不引入新的错误率或功能退化
4. 优化效果可通过 Admin 后台和日志可量化验证

### 1.3 非目标

- 不改变现有监控埋点逻辑（已由 feat-04 完成）
- 不优化音频录制、BLE 传输等 iOS 端链路
- 不要求 100% 的请求都达标（网络抖动、冷启动等偶发场景除外）
- 不引入本地推理（离线模型）方案

---

## 2. 用户与场景

**用户**：开发者 / 最终用户

**场景**：

1. 用户说完一句话停止录音后，**2 秒内**听到 AI 的第一句回复音频，整体对话感觉流畅自然。
2. 开发者在 Admin 后台对比优化前后的平均 LLM TTFT 数据，确认优化效果。
3. 开发者调整 LLM 模型或参数后，通过日志中的 `[Latency]` 行实时观察各环节变化。

---

## 3. 功能描述

### 3.1 LLM 响应速度优化

当前 LLM TTFT 达 5475ms，需从以下方向寻找优化空间：

**方向一：模型选型**
- 当前使用 `qwen-turbo`，可尝试更快的模型（如更小参数量模型或专门优化 TTFT 的推理端点）
- 评估不同模型在 TTFT 和回复质量之间的平衡

**方向二：Prompt 优化**
- 缩短系统 Prompt 长度，减少输入 Token 数
- **不减少对话历史轮数**：随着使用场景深入，上下文窗口只会增大，裁剪历史不是可持续的方向
- 当前系统 Prompt：`你是一个简洁友好的语音助手，请用中文回答，每次回复不超过50字`

**方向三：LLM 流式输出驱动 TTS（流水线核心）**
- LLM 已支持流式输出（这也是 TTFT 可被测量的原因），但当前 TTS **等待 LLM 全部生成完毕**后才开始合成
- 优化目标：LLM 流式 Token 到达时，按句子边界切分，每积累到一个完整句子就立即送入 TTS 合成，TTS 合成与 LLM 继续输出并行进行
- 效果：TTS 首包延迟从"等 LLM 全部完成"缩短为"等 LLM 第一句话输出完"，E2E 首包时间可显著压缩
- 回复文本越长，流水线收益越大；当前回复≤50字时收益有限，但为后续上下文增长预留优化空间

### 3.2 ASR 优化（次要）

当前 ASR 使用批量 Paraformer 接口，耗时 951ms。可评估：

- 是否有流式 ASR 接口可将首帧识别时间提前
- 流式 ASR 与现有 VAD 切断逻辑的兼容性

### 3.3 性能基线与回归验证

优化前后需在同等测试条件下对比：

- 每次调整后，进行至少 5 次有效语音交互
- 对比 Admin 后台的平均 LLM_TTFT 和 E2E_ttfa 指标
- 确认无新增 STT_FAIL 或 STT_EMPTY 错误

---

## 4. 约束与依赖

| 约束 | 说明 |
|------|------|
| 回复质量不可明显下降 | 优化后 AI 回复仍需语义通顺、内容完整 |
| 不破坏现有监控埋点 | LatencyRecord 结构不得简化，优化后数据仍需写入数据库 |
| DashScope API 限制 | 只能使用阿里云 DashScope 提供的模型和接口 |
| 不引入新依赖 | 优先配置级和参数级调整，不新增第三方服务 |

**依赖模块**：
- `pipecat-pipeline`（`pipecat-server/`）：优化目标模块，含 LLM 调用、TTS 流水线
- `feat-04`（已完成）：提供可量化的延迟测量基础设施

---

## 5. 验收标准

1. 5 次有效语音交互的 **平均 E2E_ttfa ≤ 2000ms**
2. 5 次有效语音交互的 **平均 LLM_TTFT ≤ 1000ms**
3. Admin 后台耗时数据正常写入，无字段缺失
4. 日志 `[Latency]` 行中不再触发 LLM_TTFT 慢请求 WARNING
5. STT 识别错误率未因修改而升高
6. 现有测试套件全部通过

---

## 6. 优先级

**P0 - 最高优**：当前 7257ms 的首包延迟已严重影响语音助手的可用性，是当前阶段最优先解决的体验问题。
