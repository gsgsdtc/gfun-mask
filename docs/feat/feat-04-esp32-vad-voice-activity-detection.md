# Feat-04: ESP32 VAD 语音活动检测

> 更新日期：2026-03-13
> 来源：用户需求
> 优先级：P1

## 1. 功能概述

### 1.1 背景

当前语音聊天功能需要用户在 iOS 界面上手动点击"开始"和"结束"按钮来控制录音，这种交互方式对于可穿戴设备场景不够自然，且增加了用户的操作负担。

ESP32 作为音频采集的前端设备，具备实时处理音频流的能力。在 ESP32 侧引入 VAD（Voice Activity Detection，语音活动检测），可以自动识别用户何时开始说话、何时停止说话，从而消除手动操控的需求，实现"开口即说、停口即止"的自然交互。

### 1.2 目标

- 在 ESP32 固件中集成 VAD 算法，实时检测麦克风输入的语音活动状态
- 检测到语音开始时，通过 BLE L2CAP 通知 iOS 端，触发聊天流程自动开始
- 检测到语音结束（静音超过阈值）时，通过 BLE 通知 iOS 端，触发聊天流程自动结束
- iOS 侧聊天界面无需用户手动操作开始/结束按钮

### 1.3 非目标

- 不在 iOS 端做 VAD（VAD 只在 ESP32 侧处理）
- 不改变音频编码格式（仍使用 Opus）
- 不引入云端 VAD 服务
- 不处理多说话人区分
- 不实现唤醒词检测（Wake Word）

## 2. 用户场景

### 场景 1: 自然发起对话

- **角色**：佩戴 VoiceMask 的用户
- **触发**：用户对着设备开口说话
- **流程**：
  1. ESP32 麦克风持续采集音频
  2. VAD 检测到语音活动开始
  3. ESP32 通过 BLE 发送"语音开始"事件给 iOS
  4. iOS 自动进入录音/聊天状态，无需用户点击任何按钮
  5. 用户完成表达后停止说话
- **结果**：iOS 端聊天流程自动启动，用户无需手动操作

### 场景 2: 自然结束对话

- **角色**：佩戴 VoiceMask 的用户
- **触发**：用户说话结束后保持沉默
- **流程**：
  1. 用户停止说话，麦克风采集到静音
  2. VAD 检测到静音持续超过设定阈值
  3. ESP32 通过 BLE 发送"语音结束"事件给 iOS
  4. iOS 自动停止录音，触发 STT/LLM/TTS 处理流程
- **结果**：聊天流程自动完成，无需用户手动点击结束

### 场景 3: 噪音环境下的误触发抑制

- **角色**：佩戴 VoiceMask 的用户（处于嘈杂环境中）
- **触发**：环境中存在非语音噪音（风声、环境音等）
- **流程**：
  1. ESP32 麦克风采集到环境噪音
  2. VAD 识别该音频为非语音，不触发"语音开始"事件
  3. 系统保持待机状态
- **结果**：不因噪音产生误触发，避免干扰用户

## 3. 验收标准

### 功能验收

- [ ] VAD 能在正常说话音量下稳定检测到语音开始，延迟不超过 200ms
- [ ] VAD 能在用户停止说话后，于 500ms～1500ms 内检测到语音结束
- [ ] iOS 侧聊天界面在收到"语音开始"事件后自动进入录音状态，无需手动点击
- [ ] iOS 侧聊天界面在收到"语音结束"事件后自动触发 STT 处理，无需手动点击
- [ ] 手动开始/结束按钮作为兜底保留，VAD 触发与手动触发不冲突
- [ ] BLE L2CAP 通道正常传输 VAD 事件帧（语音开始帧 / 语音结束帧）

### 非功能验收

- [ ] 性能：VAD 处理不影响 ESP32 主任务，音频采集不发生丢帧
- [ ] 稳定性：连续运行 30 分钟无崩溃、无内存泄漏
- [ ] 功耗：VAD 运行时不应导致明显发热或电池异常消耗（无具体预算，以实测为准）
- [ ] 算法选型：优先使用乐鑫官方提供的、适配 ESP32-S3 的 VAD 算法（如 ESP-SR 中的 AFE/VAD 组件），避免引入第三方算法带来的移植风险

## 4. 约束条件

- ESP32 计算资源有限，VAD 算法需轻量，优先采用乐鑫官方 ESP-SR 中适配 ESP32-S3 的 VAD/AFE 组件，次选其他轻量算法
- VAD 事件通过现有 BLE L2CAP 通道传输，需与协议帧格式兼容（`0xFF` pre-warm 帧 / `0xFE` 结束帧）
- 静音结束阈值（silence timeout）需可配置，默认建议 800ms
- VAD 检测必须在 ESP32 本地实时执行，不依赖网络
- 依赖 feat-01（BLE L2CAP 通道）和 feat-02（音频采集）已完成
